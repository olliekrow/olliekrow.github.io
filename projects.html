GitHub PM Architecture

GitHub Projects GitHub Actions GraphQL YAML Automation
↗ View on GitHub ↗ Read the Docs

A full org-level GitHub PM setup, built from scratch in a personal demo space before proposing it to a real organisation. The goal was to replace a scattered collection of repo-level projects with a coherent, cross-repo architecture using only native GitHub tooling.

What was built

  • A YAML issue template suite covering Epics, User Stories, Tasks, Bug Reports, and Feature Requests — each with structured fields and consistent labelling conventions.
  • Three org-level Projects v2 boards: a Roadmap (Epics on a timeline by milestone), a Team Planning board (Kanban with iteration support), and a Bug Tracker.
  • Custom fields including Start Date, End Date, and a Size field (XS–XL mapped to Fibonacci values 1/2/3/5/8) serving as the estimation proxy — no separate Story Points field needed.
  • A GitHub Action (set-dates.yml) that auto-populates Start Date on status transition to In Progress and End Date on transition to Done, using the Projects v2 GraphQL API.

Key decisions

  • Label-based filtering over issue types — GitHub issue types weren't rendering reliably in project filters, so label-based filtering (e.g. the epic label) is the more dependable workaround for Roadmap views.
  • Org-level projects only — repo-level projects lack cross-repo visibility. Org-level is the correct architecture for any team working across multiple repositories.
  • Size as estimation proxy — avoids introducing a second field and keeps the estimation process lightweight.
"Prototyping in a personal demo space before proposing org changes is how you manage stakeholder risk. Nobody likes a surprise restructure."

Kids' Mobile GDD Template

Game Design GDD Kids' Mobile Google Docs
↗ Read the Docs

A polished, reusable game design document template built specifically for kids' mobile games. The template was developed using an existing high-quality GDD as a quality benchmark, with the goal of producing something immediately usable rather than a generic starting point.

Design philosophy

  • Card-style persona blocks over side-by-side layouts — tested both approaches; card-style outperforms for readability and is easier to update without reformatting the whole document.
  • Gender-neutral language as a hard standard — descriptive names and they/them pronouns throughout. Not a stylistic preference, a firm requirement.
  • Design pillars as decision filters — each pillar is written so it can be used to reject a design proposal. A pillar that can't say no to something isn't doing its job.
  • Accessibility as a first-class concern — kids' games require no text literacy dependency, clear visual affordances, short session design, and consideration for players with motor or cognitive differences.

Template structure

  • Game Overview — elevator pitch, platform, audience, genre, tone
  • Design Pillars — 3–5 decision-filter statements
  • Core Loop — primary and secondary loops with session arc
  • Player Personas — card-style with name, age, context, motivations, and pain points
  • Progression Systems — rewards, unlocks, difficulty curve
  • Accessibility Considerations — visual, motor, cognitive
  • Out of Scope — explicit list of things the game will not do
"A GDD is not a spec. It's a thinking tool — a way of forcing decisions that would otherwise get deferred until they become expensive problems mid-development."

Project Hub Docs Site

Jekyll Just the Docs GitHub Pages GitHub Actions CI/CD
↗ View Live Site ↗ View on GitHub

A Jekyll-based documentation site using the Just the Docs theme, covering PM workflows, game design references, and onboarding material. Deployed automatically on every push to main via GitHub Actions — no manual deployment steps required.

Tech stack

  • Jekyll with the Just the Docs v0.10.0 theme via remote_theme
  • GitHub Actions workflow for automatic build and deployment
  • Structured navigation with section index files and permalink conventions
  • Hosted on GitHub Pages (free tier)

Site structure

  • GitHub PM — issue templates, project architecture, workflow automation and the GraphQL queries needed to retrieve field IDs
  • Game Design — GDD template reference and structure documentation
  • Onboarding — placeholder, to be built out
  • Roadmap — placeholder, to be built out
Built entirely with native GitHub tooling. No third-party hosting, no paid CI/CD service, no CMS — just a repo, a theme, and a workflow file.
