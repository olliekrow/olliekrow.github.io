How I work

I'm a Program Manager with a background that spans operations, game design, and technical writing. My approach is rooted in lean, native tooling — I'd rather get 90% of the value from tools a team already has than introduce a new platform that adds overhead without adding clarity.

On the PM side, I build structured systems from scratch. That means issue templates, project boards, automation workflows, and documentation — not as an end in themselves, but as scaffolding that helps teams move faster with less friction. I prototype in a personal demo space before proposing anything at the org level, because changes to shared infrastructure deserve a soft landing.

On the game design side, I write GDDs that function as thinking tools rather than spec documents. A good GDD forces decisions early — about core loop, about who the player is, about what the game refuses to compromise on. I specialise in kids' mobile games, where the design constraints are the most interesting: short sessions, no text literacy required, accessibility as a first-class concern.

I also write — about process, craft, and the things that make work worth doing. The blog is where I work through ideas that don't fit anywhere else.

Principles

01

Demo before proposing

Nobody likes a surprise restructure. Prototype the change in a safe space first. Show, don't pitch.

02

Lean tooling

The best tool is the one a team already has. Add a new platform only when the native option genuinely can't do the job.

03

Documentation as first-class work

A system nobody can onboard into is a system that will break. Good docs aren't overhead — they're the product.

04

Design pillars as filters

A design pillar isn't a vibe. It's a decision-making tool. If it can't be used to say no to something, it's not doing its job.

05

Estimation without overhead

Story points, velocity, T-shirt sizes — pick one and make it dead simple. Complexity in the estimation system defeats its own purpose.

06

Iterative refinement

The first version is a hypothesis. Ship it, learn from it, improve it. Quality is a direction, not a destination.
